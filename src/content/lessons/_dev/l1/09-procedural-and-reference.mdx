Some lessons teach the right procedure once. Others build the muscle memory to do that procedure under pressure. These two elements split that work.

## Console — pick-the-command mode, with per-shell variants

Multiple-choice variant. The learner picks the right next command, the response renders, the next step unlocks. Turns rote command memorisation into active recall.

Each step here declares per-shell `variants` (bash, PowerShell, cmd). The toggle in the Console header is the visitor's preference; pick once, every Console on this page and across the site uses the same shell. Saved to localStorage.

Wrong picks render the per-choice **rationale** under the chosen option — that's the teaching moment for look-alikes like `ping` (resolves the name, but only shows one IP, so you miss multi-A and CNAME-chained answers).

## Console — free-type mode

Same component, different per-step shape. Drop the `choices` array, supply `expected` as a string or RegExp. The learner types into a prompt that looks like a real shell. Free-type is harder than multiple-choice — the learner has to know the shape of the command, not just recognise it. Use it when finger memory is the actual skill.

<Console
  client:load
  title="mail-auth"
  prompt="$ "
  steps={[
    {
      narration: "Pull the SPF record for example.com.",
      expected: /dig\s+(\+short\s+)?(txt\s+)?example\.com(\s+txt)?/i,
      response: '"v=spf1 ip4:203.0.113.10 -all"',
      comment: "Accepts dig with optional flags in any order.",
    },
    {
      narration: "Now pull the DMARC record. Different hostname.",
      expected: /dig\s+(\+short\s+)?(txt\s+)?_dmarc\.example\.com(\s+txt)?/i,
      response: '"v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com"',
    },
  ]}
/>

## Notes on choices

- **Per-choice `rationale`** is the preferred way to teach a wrong answer. Authors should imagine the technician who picks the look-alike — what's the specific trap they need to learn? `ping` resolves the name but hides multi-A/CNAME; `Test-NetConnection` is reachability, not record-set; `whois` is registrar metadata, not zone records. The rationale shows beneath the picked choice when the learner submits the wrong one.
- **Per-step `variants`** drive the shell toggle. Authors don't need to provide all three (bash + PowerShell + cmd) — if only bash + PowerShell are defined, cmd falls back to the first declared variant so the step is still usable.
- Steps that aren't shell-specific (vendor CLI like `cpcmd`, `gam`, `Get-CIPPUser`) skip `variants` and use top-level `choices` / `expected` / `response` directly — the shell toggle doesn't appear if no step has variants.

## Runbook

Same `<Step>` children as `StepThrough`, but everything visible at once and no Continue button. The mental model is "look up", not "learn". Useful for on-call playbooks, offboarding checklists, anything the learner already knows but needs a structured place to follow.

<Runbook title="M365 user offboarding" audience="Helpdesk L1">
  <Step title="Block sign-in">
    In Entra → Users → the leaver, set **Block sign-in** to Yes. Revokes existing tokens within an hour.
  </Step>
  <Step title="Convert mailbox to shared">
    Exchange Admin Center → Recipients → Mailboxes → the leaver → Convert to shared mailbox. Frees the licence without losing mail.
  </Step>
  <Step title="Forward mail to manager">
    On the now-shared mailbox, add the manager's UPN to **Mailbox delegation → Full access**. Set an out-of-office that names the manager.
  </Step>
  <Step title="Remove licences">
    Back in Entra → the leaver → Licences → remove the M365 SKU. Wait 24 hours before deleting the account.
  </Step>
  <Step title="Hand over OneDrive">
    SharePoint Admin → Active sites → the leaver's OneDrive → Permissions → grant the manager. OneDrive is auto-deleted 30 days after account deletion unless retention is configured.
  </Step>
</Runbook>

Runbooks share the same numbered-bubble styling as StepThrough, so authors can move content between the two without rewriting it.
